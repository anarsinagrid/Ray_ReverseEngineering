# Why Ray Injects Tracing Exactly Once in `RemoteFunction`

Ray treats tracing injection as a lazy, one-time initialization step. Re-injecting on every `.remote()` call would turn a cheap dispatch path into a repeatedly mutating, thread-sensitive wrapper stack with memory, correctness, and performance penalties.

The guard in `RemoteFunction._remote()` freezes the dispatch object after its first use ([remote_function.py#L341-L346](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
        with self._inject_lock:
            if self._function_signature is None:
                self._function = _inject_tracing_into_function(self._function)
                self._function_signature = ray._common.signature.extract_signature(
                    self._function
                )
```

**Core Invariant:** The `RemoteFunction` object must act as an immutable, read-mostly dispatch handle after its first invocation.

## 1. What Breaks Without One-Time Injection

**Unbounded Closure Chain (Memory Leak):**
`_inject_tracing_into_function()` wraps `self._function` and replaces it ([tracing_helper.py#L326-L363](https://github.com/ray-project/ray/tree/master/python/ray/util/tracing/tracing_helper.py))

```python
def _inject_tracing_into_function(function):
    if not _is_tracing_enabled():
        return function

    function.__signature__ = _add_param_to_signature(...)

    @wraps(function)
    def _function_with_tracing(*args, **kwargs):
        # ... tracing setup ...
        return function(*args, **kwargs)

    return _function_with_tracing
```

Calling this on every `.remote()` would bury the function N closures deep, leaking memory. `_add_param_to_signature` has an idempotency guard ([tracing_helper.py#L115-L124](https://github.com/ray-project/ray/tree/master/python/ray/util/tracing/tracing_helper.py))

```python
def _add_param_to_signature(function, new_param):
    # If new_param is already in signature, do not add it again.
    if any(param.name == new_param.name for param in old_sig_list_repr):
        return old_sig
```

but it does not protect the function itself from being recursively re-wrapped.

**Race Conditions in Multi-Threaded Drivers:**
`_inject_lock` is established in `__init__` ([remote_function.py#L152-L153](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
        # Guards trace injection to enforce exactly once semantics
        self._inject_lock = Lock()
```

Without the lock, concurrent dispatch threads would race to overwrite `self._function` and `self._function_signature`, leaving the object in a misaligned state.

**Stale Pickled Functions on Workers:**
After injection, the function is pickled for export ([remote_function.py#L366-L372](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
            self._pickled_function = pickle_dumps(
                self._function,
                f"Could not serialize the function {self._function_descriptor.repr}",
            )
            self._last_export_cluster_and_job = worker.current_cluster_and_job
            worker.function_actor_manager.export(self)
```

If the driver re-wrapped the function on each call, the local closure would diverge from the exported version running on workers.

**Hot-Path Overhead:**
`extract_signature()` uses the expensive `inspect.signature` call ([signature.py#L58-L79](https://github.com/ray-project/ray/tree/master/python/ray/_common/signature.py))

```python
def extract_signature(func, ignore_first=False):
    signature_parameters = list(get_signature(func).parameters.values())
    # ...
    return signature_parameters
```

The result is required by `flatten_args()` on every task submission ([remote_function.py#L480-L485](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
            else:
                list_args = ray._common.signature.flatten_args(
                    self._function_signature, args, kwargs
                )
```

Re-extracting per call would crush submission throughput.

## 2. Lock and Sentinel: The Safest Tradeoff

The lock-and-sentinel delays tracing overhead until first use. Tracing requires the Ray runtime, so it cannot run eagerly at `@ray.remote` decoration time ([remote_function.py#L339-L340](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
        # We cannot do this when the function is first defined, because we need
        # ray.init() to have been called when this executes
```

When state is serialized, `__getstate__` drops the unpicklable Lock but preserves `_function_signature` so deserialized objects skip the injection block immediately ([remote_function.py#L180-L188](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
    def __getstate__(self):
        attrs = self.__dict__.copy()
        del attrs["_inject_lock"]
        return attrs

    def __setstate__(self, state):
        self.__dict__.update(state)
        self.__dict__["_inject_lock"] = Lock()
```

## 3. Failure Semantics & Abstraction Leaks

**Injection Failure Recovery:**
If injection raises an exception, `_function_signature` remains `None`. Subsequent calls re-attempt initialization without retaining corrupted state.

**Invisible Mutation:**
`RemoteFunction` appears immutable post-decoration but mutates itself on first `.remote()`. Introspecting `self._function` across this boundary reveals different callable objects.

## Summary

First-call injection isolates tracing setup from recurring execution. By guarding with a lock and sentinel, Ray pays the reflection and wrapping cost exactly once, keeping `RemoteFunction` as a stable dispatch handle.
