# Why Ray Uses `num_cpus=0` + `_force_on_current_node()` for Logging Actors

These two settings work together to place lightweight I/O-bound sidecar processes on the driver node without competing with compute tasks.

**Core Invariant:** Sidecar actors must bypass CPU admission control (so they don't starve compute), while still running on a specific node (so they have access to local filesystem paths and secrets). `num_cpus=1` steals from training; no affinity constraint scatters the sidecar across the cluster.

## 1. Bypassing Admission Control with `num_cpus=0`

Ray CPU resources are logical, not physical ([resources.rst#L23-L28](https://github.com/ray-project/ray/tree/master/doc/source/ray-core/scheduling/resources.rst))

```rst
Ray resources are **logical** and don't need to have 1-to-1 mapping with physical resources.
They are mainly used for admission control during scheduling.
```

Setting `num_cpus=0` makes the actor invisible to resource-based scheduling — it can run on any node without consuming a CPU slot. The W&B logging actor uses this ([wandb.py#L705-L720](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
            self._remote_logger_class = ray.remote(
                num_cpus=0,
                **_force_on_current_node(),
                runtime_env={"env_vars": env_vars},
                max_restarts=-1,
                max_task_retries=-1,
            )(self._logger_actor_cls)

        self._trial_queues[trial] = Queue(
            actor_options={"num_cpus": 0, **_force_on_current_node(), "max_restarts": -1, ...}
        )
```

Tune's internal `TunerInternal` and progress reporter queue use the same pattern ([tuner.py#L130-L132](https://github.com/ray-project/ray/tree/master/python/ray/tune/tuner.py))

```python
                self._remote_tuner = _force_on_current_node(
                    ray.remote(num_cpus=0)(TunerInternal)
                ).remote(**kwargs)
```

([progress_reporter.py#L1557-L1560](https://github.com/ray-project/ray/tree/master/python/ray/tune/progress_reporter.py))

```python
            string_queue = Queue(
                actor_options={"num_cpus": 0, **_force_on_current_node(None)}
            )
```

This separation of admission control from physical placement mirrors patterns in [Apache Mesos](http://mesos.apache.org/documentation/latest/architecture/) and Borg, where system daemons are exempt from resource accounting used for user workloads.

## 2. Pinning to the Driver via `_force_on_current_node()`

A zero-resource actor triggers a special scheduler behavior: it is placed on a random node ([index.rst#L70-L71](https://github.com/ray-project/ray/tree/master/doc/source/ray-core/scheduling/index.rst))

```rst
Currently Ray handles actors that don't require any resources (i.e., ``num_cpus=0`` with no
other resources) specially by randomly choosing a node in the cluster without considering
resource utilization.
```

This is confirmed in C++ ([cluster_resource_scheduler.cc#L159-L164](https://github.com/ray-project/ray/tree/master/src/ray/raylet/scheduling/cluster_resource_scheduler.cc))

```cpp
  // The zero cpu actor is a special case.
  if (actor_creation && resource_request.IsEmpty() &&
      !IsHardNodeAffinitySchedulingStrategy(scheduling_strategy)) {
    return scheduling_policy_->Schedule(resource_request, SchedulingOptions::Random());
  }
```

Random placement breaks W&B logging actors, which need to access the driver's local filesystem and secrets.

`_force_on_current_node()` overrides this by reading the driver's node ID and applying a hard `NodeAffinitySchedulingStrategy` ([node.py#L48-L69](https://github.com/ray-project/ray/tree/master/python/ray/air/util/node.py))

```python
def _force_on_current_node(remote_func_or_actor_class=None):
    """Schedule a remote function or actor class on the current node."""
    current_node_id = ray.get_runtime_context().get_node_id()
    return _force_on_node(current_node_id, remote_func_or_actor_class)
```

([node.py#L35-L45](https://github.com/ray-project/ray/tree/master/python/ray/air/util/node.py))

```python
    scheduling_strategy = ray.util.scheduling_strategies.NodeAffinitySchedulingStrategy(
        node_id=node_id, soft=False
    )
    options = {"scheduling_strategy": scheduling_strategy}
    if remote_func_or_actor_class is None:
        return options
    return remote_func_or_actor_class.options(**options)
```

`soft=False` makes this a hard constraint. If the target node dies, the actor fails rather than being rescheduled elsewhere into an inconsistent state ([index.rst#L97-L104](https://github.com/ray-project/ray/tree/master/doc/source/ray-core/scheduling/index.rst))

```rst
The ``soft`` flag specifies whether the task or actor is allowed to run somewhere else
if the specified node doesn't exist or is infeasible. If ``soft`` is True, the task or actor
will be scheduled onto a different feasible node. Otherwise, it will fail with
``TaskUnschedulableError`` or ``ActorUnschedulableError``.
```

The C++ comment confirms that hard node affinity bypasses the zero-resource random dispersal rule entirely: _"except for HARD node affinity scheduling policy"_.

## Summary

`num_cpus=0` and `_force_on_current_node()` address two separate problems — one is about admission control, the other is about placement. Each setting alone fails: `num_cpus=0` without affinity scatters the actor randomly; affinity without `num_cpus=0` starves CPU slots. Together they produce a zero-cost, driver-pinned sidecar ([node.py#L53-L56](https://github.com/ray-project/ray/tree/master/python/ray/air/util/node.py))
