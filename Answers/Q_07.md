# Why Ray Uses Different Argument Formatting for Cross-Language Functions

Ray has two separate argument formatting paths: one for Python-to-Python execution and one for cross-language calls. A unified format would either force non-Python runtimes to parse Python-specific serialization, or strip Python of native calling conventions like kwargs.

**Core Invariant:** Arguments delivered to a non-Python worker must be decodable by that runtime using language-neutral formats like [MessagePack](https://msgpack.org/index.html), without any reliance on Python-specific shims.

## 1. The Python Path

For Python-to-Python calls, `RemoteFunction._remote` uses `flatten_args()`, which produces a flat list pairing arguments with `b"__RAY_DUMMY__"` sentinels ([signature.py#L104-L137](https://github.com/ray-project/ray/tree/master/python/ray/_common/signature.py))

```python
def flatten_args(signature_parameters, args, kwargs):
    """Validates the arguments against the signature and flattens them."""
    list_args = []
    for arg in args:
        list_args += [DUMMY_TYPE, arg]
    for keyword, arg in kwargs.items():
        list_args += [keyword, arg]
    return list_args
```

The receiving end uses `recover_args` to reconstruct positional and keyword arguments ([signature.py#L140-L163](https://github.com/ray-project/ray/tree/master/python/ray/_common/signature.py))

```python
def recover_args(flattened_args):
    args = []
    kwargs = {}
    for name_index in range(0, len(flattened_args), 2):
        name, arg = flattened_args[name_index], flattened_args[name_index + 1]
        if name == DUMMY_TYPE:
            args.append(arg)
        else:
            kwargs[name] = arg
    return args, kwargs
```

Java workers calling Python functions mirror this explicitly, prepending `PYTHON_DUMMY_TYPE` so Python's receiver interprets the argument list correctly ([ArgumentsBuilder.java#L22-L70](https://github.com/ray-project/ray/tree/master/java/runtime/src/main/java/io/ray/runtime/task/ArgumentsBuilder.java))

```java
  private static final NativeRayObject PYTHON_DUMMY_TYPE =
      ObjectSerializer.serialize("__RAY_DUMMY__".getBytes());

  public static List<FunctionArg> wrap(Object[] args, Language language) {
    // ...
    if (language == Language.PYTHON) {
      ret.add(FunctionArg.passByValue(PYTHON_DUMMY_TYPE));
    }
    // ...
  }
```

## 2. The Cross-Language Path

When `_is_cross_language` is true, Ray detects the language at initialization ([remote_function.py#L148-L156](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
        self._is_cross_language = language != Language.PYTHON
```

and routes to `cross_language._format_args()` instead of `flatten_args` ([remote_function.py#L477-L485](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
        if self._is_cross_language:
            list_args = cross_language._format_args(worker, args, kwargs)
        else:
            list_args = ray._common.signature.flatten_args(
                self._function_signature, args, kwargs
            )
```

`_format_args` passes positional args through as-is and rejects kwargs with a `TypeError` ([cross_language.py#L83-L103](https://github.com/ray-project/ray/tree/master/python/ray/cross_language.py))

```python
def _format_args(worker, args, kwargs):
    if not worker.load_code_from_local:
        raise ValueError("Cross language feature needs --load-code-from-local to be set.")
    if kwargs:
        raise TypeError(
            f"Cross language remote functions does not support kwargs, "
            f"kwargs:{str(kwargs)}."
        )
    return args
```

Java and C++ have no kwargs concept, so allowing them would create a false interface contract.

## 3. Serialization Metadata Enforcement

After argument formatting, `prepare_args_internal` in `_raylet.pyx` enforces that cross-language arguments carry valid metadata types ([_raylet.pyx#L844-L851](https://github.com/ray-project/ray/tree/master/python/ray/_raylet.pyx))

```python
            metadata = serialized_arg.metadata
            if language != Language.PYTHON:
                metadata_fields = metadata.split(b",")
                if metadata_fields[0] not in [
                        ray_constants.OBJECT_METADATA_TYPE_CROSS_LANGUAGE,
                        ray_constants.OBJECT_METADATA_TYPE_RAW,
                        ray_constants.OBJECT_METADATA_TYPE_ACTOR_HANDLE]:
                    raise TypeError(...)
```

The valid type identifiers are defined in [ray_constants.py#L344-L356](https://github.com/ray-project/ray/tree/master/python/ray/_private/ray_constants.py)

```python
OBJECT_METADATA_TYPE_CROSS_LANGUAGE = b"XLANG"
OBJECT_METADATA_TYPE_PYTHON = b"PYTHON"
OBJECT_METADATA_TYPE_RAW = b"RAW"
OBJECT_METADATA_TYPE_ACTOR_HANDLE = b"ACTOR_HANDLE"
```

and mirrored in Java ([ObjectSerializer.java#L66-L73](https://github.com/ray-project/ray/tree/master/java/runtime/src/main/java/io/ray/runtime/object/ObjectSerializer.java))

```java
  public static final byte[] OBJECT_METADATA_TYPE_CROSS_LANGUAGE = "XLANG".getBytes();
  public static final byte[] OBJECT_METADATA_TYPE_RAW = "RAW".getBytes();
  public static final byte[] OBJECT_METADATA_TYPE_ACTOR_HANDLE = "ACTOR_HANDLE".getBytes();
```

Python-to-Python calls fall back to the `PYTHON` (pickle5) type, which non-Python runtimes cannot decode ([serialization.py#L655-L677](https://github.com/ray-project/ray/tree/master/python/ray/_private/serialization.py))

```python
        if python_objects:
            metadata = ray_constants.OBJECT_METADATA_TYPE_PYTHON
            pickle5_serialized_object = self._serialize_to_pickle5(metadata, python_objects)
```

Cross-language tests validate that only msgpack-compatible types transfer between runtimes ([test_cross_language_invocation.py#L42-L47](https://github.com/ray-project/ray/tree/master/java/test/src/main/resources/test_cross_language_invocation.py))

```python
        r = ray.cross_language.java_function(
            "io.ray.test.CrossLanguageInvocationTest", "returnInputIntArray"
        ).remote([1, 2, 3])
        assert ray.get(r) == [1, 2, 3]
```

## Summary

![Q7](../Images/Q7.png)

Rather than building a universal format that works poorly in every language, Ray forks its serialization at the submission layer. Cross-language functions use a `lambda *args, **kwargs: None` stub to prevent Python's signature inspection from running ([cross_language.py#L14-L46](https://github.com/ray-project/ray/tree/master/python/ray/cross_language.py))

```python
def java_function(class_name, function_name):
    """Define a Java function."""
    return RemoteFunction(
        Language.JAVA,
        lambda *args, **kwargs: None,
        JavaFunctionDescriptor(class_name, function_name, ""),
        {},
    )
```

This makes it explicit at initialization that cross-language functions bypass Python's calling conventions entirely.
