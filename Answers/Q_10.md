# Why Ray Uses `_last_export_cluster_and_job` for Deduplication

Exporting a Python function to remote workers means serializing its bytecode and pushing it through GCS. Doing this on every `.remote()` call would saturate the control plane. Ray deduplicates exports using an internal sentinel that tracks which `(cluster, job)` the function was last exported to, as described in the [Ray architecture paper (OSDI '18)](../References/osdi18-moritz.pdf) and drawn from closure-shipping strategies in [Spark (NSDI '12)](../References/nsdi12-final138.pdf).

**Core Invariant:** A function's bytecode should be exported once per `(cluster, job)` context, not once per invocation.

## 1. Deduplicating Exports

When `.remote()` is called, `RemoteFunction._remote()` checks whether the current `(cluster, job)` matches the last export ([remote_function.py#L349-L372](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
        if (
            not self._is_cross_language
            and self._last_export_cluster_and_job != worker.current_cluster_and_job
        ):
            self._function_descriptor = PythonFunctionDescriptor.from_function(...)
            self._pickled_function = pickle_dumps(...)

            self._last_export_cluster_and_job = worker.current_cluster_and_job
            worker.function_actor_manager.export(self)
```

If they match, the export is skipped entirely. The sentinel is described in the docstring of `RemoteFunction` ([remote_function.py#L81-L88](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
        _last_export_cluster_and_job: A pair of the last exported cluster
            and job to help us to know whether this function was exported.
            This is an imperfect mechanism used to determine if we need to
            export the remote function again. It is imperfect in the sense that
            the actor class definition could be exported multiple times by
            different workers.
```

The result is O(1) GCS writes per function definition per job, not O(N) per invocation.

## 2. Worker-Side Fetch and Registration

When a worker encounters a task for an unfamiliar function, `FunctionActorManager` fetches and registers it from GCS ([function_manager.py#L283-L349](https://github.com/ray-project/ray/tree/master/python/ray/_private/function_manager.py))

```python
    def fetch_and_register_remote_function(self, key):
        """Import a remote function."""
        remote_function_info = self.fetch_registered_method(key)
        if not remote_function_info:
            return False
        # ...
        with self.lock:
            self._num_task_executions[function_id] = 0
            try:
                function = pickle.loads(serialized_function)
            except Exception:
                # ... error handling, placeholder function ...
            else:
                function.__module__ = module
                self._function_execution_info[function_id] = FunctionExecutionInfo(
                    function=function, function_name=function_name, max_calls=max_calls
                )
        return True
```

On subsequent tasks for the same function, `get_execution_info` short-circuits via a local cache check ([function_manager.py#L351-L390](https://github.com/ray-project/ray/tree/master/python/ray/_private/function_manager.py)) — if `function_id` is already in `_function_execution_info`, no GCS round-trip occurs.

The driver-side export is handled by `export()`, which checks GCS before writing ([function_manager.py#L214-L261](https://github.com/ray-project/ray/tree/master/python/ray/_private/function_manager.py))


## 3. Failure Modes

**Stale `_last_export_cluster_and_job`:**
If a `RemoteFunction` object is retained across `ray.init()` / `ray.shutdown()` cycles, the sentinel points to the old cluster. The function won't re-export on the new cluster until the `current_cluster_and_job` changes. The setup hook export path shows the GCS storage step ([function_manager.py#L179-L212](https://github.com/ray-project/ray/tree/master/python/ray/_private/function_manager.py))

```python
    def export_setup_func(self, setup_func, timeout=None):
        """Export the setup hook function and return the key."""
        pickled_function, function_to_run_id, key = build_setup_hook_export_entry(...)
        self._worker.gcs_client.internal_kv_put(
            key, pickle.dumps({"job_id": ..., "function_id": ..., "function": pickled_function}),
            True, ray_constants.KV_NAMESPACE_FUNCTION_TABLE,
        )
        return key
```

**Missing Dependencies:**
Ray exports the function bytecode but not its dependencies. If a dependency is absent in the worker environment, the worker gets a `ModuleNotFoundError` at deserialization time. The `runtime_env` mechanism in `remote_function.py` and `actor.py` (not `function_manager.py`) handles dependency shipping.

## Summary

![Q10](../Images/Q10.png)

`_last_export_cluster_and_job` separates export from invocation. The function definition is pushed to GCS once; every subsequent call reuses the cached version on workers. This keeps the GCS control plane from becoming a bottleneck at scale.
