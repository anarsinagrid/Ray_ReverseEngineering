# Why Ray Uses a Separate `_WandbLoggingActor` Per Trial

Ray's W&B integration uses a dedicated sidecar actor per trial to satisfy W&B's process isolation requirements and Ray's constraints against unmanaged multiprocessing.

**Core Invariant:** Per-trial W&B logging must be isolated. Trials must not observe or contaminate each other's metrics, run state, or file handles.

## 1. Process Isolation Is Required

W&B's `wandb.run` state operates as a process-local singleton. The `_WandbLoggingActor` docstring states this directly: _"Wandb assumes that each trial's information should be logged from a separate process"_ ([wandb.py#L392-L405](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
class _WandbLoggingActor:
    """
    Wandb assumes that each trial's information should be logged from a
    separate process. We use Ray actors as forking multiprocessing
    processes is not supported by Ray...
    """
```

If multiple trials called `wandb.init()` in the same process, their states would collide.

Standard Python `multiprocessing` is off the table:

- **`fork`:** Unsafe. Ray runs background threads for Object Store and GCS. Forking duplicates only the calling thread, causing deadlocks in the child.
- **`spawn`:** Unreliable. It starts an empty Python interpreter via `pickle`, which conflicts with unpicklable Ray internals and ML framework objects.

Ray actors provide the required isolation. Because the actor is a separate process, W&B can set `WANDB_START_METHOD=thread` safely ([wandb.py#L430-L433](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
    def run(self):
        # Since we're running in a separate process already, use threads.
        os.environ["WANDB_START_METHOD"] = "thread"
        run = self._wandb.init(*self.args, **self.kwargs)
```

See also: [Ray architecture paper](../References/osdi18-moritz.pdf).

## 2. Dedicated Queues Decouple Compute from I/O Latency

W&B network uploads are I/O-bound. Running them inline would stall the training loop.

`WandbLoggerCallback` creates one actor and one queue per trial ([wandb.py#L626-L631](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
        self._trial_logging_actors: Dict[
            "Trial", ray.actor.ActorHandle[_WandbLoggingActor]
        ] = {}
        self._trial_queues: Dict["Trial", Queue] = {}
```

Instantiated via `_start_logging_actor` ([wandb.py#L713-L730](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
        self._trial_queues[trial] = Queue(actor_options={"num_cpus": 0, **_force_on_current_node(), ...})
        self._trial_logging_actors[trial] = self._remote_logger_class.remote(
            logdir=trial.local_path, queue=self._trial_queues[trial], ...
        )
```

The actor consumes the queue in a loop ([wandb.py#L438-L463](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
        while True:
            item_type, item_content = self.queue.get()
            if item_type == _QueueItem.END:
                break
            # ...
            self._wandb.log(log, step=log.get(TRAINING_ITERATION))
        self._wandb.finish()
```

enqueuing lightweight `_QueueItem.RESULT` descriptors instead of calling `wandb.log()` synchronously from the training loop.

At experiment end, `on_experiment_end` waits for all uploads to flush ([wandb.py#L787-L790](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
    def on_experiment_end(self, trials, **info):
        """Wait for the actors to finish their call to `wandb.finish`."""
        self._cleanup_logging_actors(timeout=self._upload_timeout, kill_on_timeout=True)
```

## 3. Pinning & Fault Tolerance

The actor is configured with `max_restarts=-1` and `max_task_retries=-1` so transient W&B API errors do not permanently lose trial telemetry ([wandb.py#L700-L711](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
            self._remote_logger_class = ray.remote(
                num_cpus=0,
                **_force_on_current_node(),
                runtime_env={"env_vars": env_vars},
                max_restarts=-1,
                max_task_retries=-1,
            )(self._logger_actor_cls)
```

`_force_on_current_node()` pins the actor to the Tune driver node via `NodeAffinitySchedulingStrategy` ([node.py#L48-L69](https://github.com/ray-project/ray/tree/master/python/ray/air/util/node.py))

```python
def _force_on_current_node(remote_func_or_actor_class=None):
    """Schedule a remote function or actor class on the current node."""
    current_node_id = ray.get_runtime_context().get_node_id()
    return _force_on_node(current_node_id, remote_func_or_actor_class)
```

This ensures the actor has direct access to local secrets and filesystem paths ([wandb.py#L701-L704](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
            if WANDB_ENV_VAR in os.environ:
                env_vars[WANDB_ENV_VAR] = os.environ[WANDB_ENV_VAR]
```

The `_logger_actor_cls` class attribute enables testing via mock substitution ([wandb.py#L583](https://github.com/ray-project/ray/tree/master/python/ray/air/integrations/wandb.py))

```python
    _logger_actor_cls = _WandbLoggingActor
```

allowing `_MockWandbLoggingActor` to replace the real implementation in tests ([mocked_wandb_integration.py#L86-L108](https://github.com/ray-project/ray/tree/master/python/ray/air/tests/mocked_wandb_integration.py))

```python
class _MockWandbLoggingActor(_WandbLoggingActor):
    _mock_wandb_api_cls = _MockWandbAPI
```

## 4. Failure Semantics & Scale Tradeoffs

**Actor Crash and Run Duplication:**
If the actor dies mid-trial, Ray restarts it with a fresh `__init__`, losing unsaved memory. W&B registers this as a new "resumed" run segment. Process continuity does not equal state continuity.

**Queue Backpressure:**
The queue has no synchronous backpressure against the training loop. Under sustained W&B latency, the queue can grow unbounded.

**Driver Resource Cost:**
1,000 parallel trials spawn 1,000 sidecar processes on the driver node, taxing OS tables and file descriptor limits. This is the direct cost of per-trial isolation.
