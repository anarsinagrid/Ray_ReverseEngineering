# Why Ray Uses `max_calls=1` for GPU Tasks by Default

GPU tasks allocate VRAM through third-party frameworks (TensorFlow, PyTorch) in ways that bypass Python's garbage collector and often survive task completion.

**Core Invariant:** The scheduler's view of GPU availability must reflect actual physical VRAM state. Ignoring leaked VRAM causes out-of-memory crashes on the next allocation.

## 1. Process Termination Is the Only Reliable VRAM Release

Ray's documentation states the problem directly ([accelerators.rst#L718-L724](https://github.com/ray-project/ray/tree/master/doc/source/ray-core/scheduling/accelerators.rst))

```rst
Currently, when a worker executes a task that uses a GPU (e.g.,
through TensorFlow), the task may allocate memory on the GPU and may not release
it when the task finishes executing. This can lead to problems the next time a
task tries to use the same GPU. To address the problem, Ray disables the worker
process reuse between GPU tasks by default, where the GPU resources is released after
the task process exits. Since this adds overhead to GPU task scheduling,
you can re-enable worker reuse by setting ``max_calls=0``
```

PyTorch's [CUDA caching allocator](https://pytorch.org/docs/stable/notes/cuda.html) pools memory and does not release VRAM blocks to the OS until the process exits. If a worker returns to the idle pool (`max_calls=0`), the scheduler thinks the GPU is free, but the next task hits OOM on the cached allocations from the previous run.

Ray intercepts task compilation: if `num_gpus > 0`, it sets `max_calls = 1` ([remote_function.py#L106-L120](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
        # When gpu is used, set the task non-recyclable by default.
        num_gpus = self._default_options.get("num_gpus") or 0
        if (
            num_gpus > 0 and self._default_options.get("max_calls", None) is None
        ) or any(
            [s in (self._default_options.get(s) or {}) for s in ["nsight", "rocprof-sys"]]
        ):
            self._default_options["max_calls"] = 1
```

When the worker hits the `max_calls` limit, it exits ([_raylet.pyx#L2219-L2228](https://github.com/ray-project/ray/tree/master/python/ray/_raylet.pyx))


Killing the OS process destroys the CUDA context and flushes all allocations. This is verified in `test_max_calls_releases_resources` ([test_basic.py#L159-L172](https://github.com/ray-project/ray/tree/master/python/ray/tests/test_basic.py))

```python
def test_max_calls_releases_resources(shutdown_only):
    ray.init(num_cpus=2, num_gpus=1)

    @ray.remote(num_cpus=1, num_gpus=1, max_calls=1, max_retries=0)
    def f():
        return [g.remote()]

    for i in range(10):
        ray.get(f.remote())  # Hangs without max_calls=1.
```

_(CPU tasks can leak state too, but Ray reuses CPU workers by default and treats `max_calls` as an opt-in control there.)_

## 2. Scale Penalties & Escape Hatches

**Throughput cost:**
Each GPU task spawns a new OS process and repeats Python + CUDA initialization. There is no warm worker pooling.

**Cache destruction:**
Model compilation optimizations are lost per invocation. Users who manage VRAM cleanup manually (e.g., via `torch.cuda.empty_cache()`) can opt out with `max_calls=0` ([accelerators.rst#L722-L736](https://github.com/ray-project/ray/tree/master/doc/source/ray-core/scheduling/accelerators.rst))

```rst

    # GPU resource leakage.

    def leak_gpus():
        # ...
```

## 3. Abstraction Leaks

**Scheduler Blind Spot:**
After worker exit, there is a window where a GPU is not yet accepting tasks but appears available to the scheduler. Concurrent tasks may queue against this slot.

**VRAM Overcommit:**
`num_gpus=1` controls scheduling slot placement, not physical VRAM partitioning. Two tasks scheduled to different logical slots on the same device can still exhaust total VRAM.

## Summary

Process death is the only mechanism Ray can rely on to guarantee VRAM reclamation. The throughput cost of restarting a worker per GPU task is cheaper than the failure cost of an OOM crash from a leaked CUDA context.
