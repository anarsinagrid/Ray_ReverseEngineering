# Why Ray Enforces `actor.method.remote()` Instead of `actor.method()`

Ray's `.remote()` is a semantic boundary between local Python execution and distributed task submission. Without it, every method call on an actor handle would be ambiguous — is it local execution or remote dispatch?

**Core Invariant:** Method invocation on an actor handle is an asynchronous dispatch to a remote process, not local execution. `actor.method()` would conflate these, destroying the caller's ability to reason about execution location and return types.

## 1. Actor Methods Are Submission Queues, Not Direct Execution

Accessing `actor.method` returns an `ActorMethod` wrapper that defers execution. The purpose is explicit: _"Create objects to wrap method invocations. This is done so that we can invoke methods with `actor.method.remote()` instead of `actor.method()`."_ ([actor.py#L822-L825](https://github.com/ray-project/ray/tree/master/python/ray/actor.py))

The method never runs locally. It routes to the dedicated worker process started at `ActorClass.remote()` time, consistent with [actor model semantics](References/osdi18-moritz.pdf) and Ray's [invocation docs](https://docs.ray.io/en/latest/ray-core/actors.html).

When `actor.method.remote(*args)` is called, the wrapper routes through `_remote()` to `_actor_method_call` ([actor.py#L1024](https://github.com/ray-project/ray/tree/master/python/ray/actor.py)), which packages the task and calls `core_worker.submit_actor_task()`, returning an `ObjectRef` immediately ([actor.py#L2411-L2427](https://github.com/ray-project/ray/tree/master/python/ray/actor.py)).

The `TaskSpecification` is submitted asynchronously ([task-lifecycle.rst#L47](https://github.com/ray-project/ray/tree/master/doc/source/ray-core/internals/task-lifecycle.rst))

```rst
4. The TaskSpecification is `submitted` to `NormalTaskSubmitter` asynchronously.
   This means the `.remote()` call returns immediately and the task is scheduled
   and executed asynchronously.
```

## 2. Preventing Silent Semantic Failures

A standard Python call returns a value. A Ray actor call returns a future (`ObjectRef`).

If `actor.method()` submitted the task automatically, the caller would get an `ObjectRef` where Python semantics expect a value. `result = actor.method()` followed by `result + 1` would fail in a non-obvious way.

Ray blocks direct invocations at the API level. `ActorMethod.__call__` raises a `TypeError` at the call site ([actor.py#L895-L900](https://github.com/ray-project/ray/tree/master/python/ray/actor.py))

The same guard exists in the Ray Client path ([common.py#L550-L555](https://github.com/ray-project/ray/tree/master/python/ray/util/client/common.py))

```python
    def __call__(self, *args, **kwargs):
        raise TypeError(
            "Actor methods cannot be called directly. Instead "
            f"of running 'object.{self._method_name}()', try "
            f"'object.{self._method_name}.remote()'."
        )
```

## 3. Sequential Consistency & Execution Ordering

Actors manage state on a single worker process. Ray serializes all method calls in submission order ([actors.rst#L184](https://github.com/ray-project/ray/tree/master/doc/source/ray-core/actors.rst))

```rst
Methods called on different actors execute in parallel, and methods called on the
same actor execute serially in the order you call them.
```

The `.remote()` call syntactically separates "submit to a remote queue" from "execute now". Removing it would make the ordering guarantee invisible to the caller.

## 4. Composability Requires Deferral

Returning an `ActorMethod` wrapper instead of executing immediately enables additional configuration:

- **Per-call options:** `actor.method.options(...)` injects settings before `.remote()` ([actor.py#L680-L709](https://github.com/ray-project/ray/tree/master/python/ray/actor.py))

```python
    valid_kwargs = [
        "num_returns",
        "concurrency_group",
        "max_task_retries",
        # ...
    ]
```

- **DAG compilation:** `actor.method.bind(...)` builds `ClassMethodNode` IR for Compiled Graphs instead of dispatching a live task ([actor.py#L815-L820](https://github.com/ray-project/ray/tree/master/python/ray/actor.py))

Auto-invocation would eliminate both.

## Summary

The `.remote()` requirement enforces location transparency, asynchrony, serialized ordering, and DAG composability as explicit, visible contract points.

Type completion in IDEs relies on placeholder classes like `_RemoteMethodNoArgs` ([actor.py#L90-L100](https://github.com/ray-project/ray/tree/master/python/ray/actor.py))

```python
class _RemoteMethodNoArgs(Generic[_Ret]):
    def remote(self) -> "ObjectRef[_Ret]":
        ...

    def bind(self) -> Any:
        ...
```

which provide return-type information without affecting runtime behavior.
