# Why the Task / Actor Split Is Fundamental

Ray separates tasks and actors because they have incompatible execution requirements: stateless scaling vs. stateful locality. The split is introduced in the [Ray architecture paper (OSDI '18)](References/osdi18-moritz.pdf).

If functions could hold mutable state, retry-on-failure would require checkpointing and rollback at the task layer. Ray keeps tasks stateless to allow zero-overhead re-execution, the same approach taken by [MapReduce](References/mapreduce.pdf) and [Spark (NSDI '12)](References/nsdi12-final138.pdf). For workloads that genuinely need process-local state (parameter servers, simulators), Ray uses actors with dedicated worker processes and Actor model semantics (Hewitt, Agha).

**Core Invariant:** Every task must be safely re-executable from any compatible worker with no dependency on prior execution history.

## 1. Worker Process Model: Pooled vs. Dedicated

Stateless tasks use reusable, pooled workers. The Raylet's `FindAndPopIdleWorker()` reuses idle processes across compatible tasks. `max_calls` controls when a worker must exit to force resource cleanup ([remote_function.py#L218-L225](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
            max_calls: This specifies the
                maximum number of times that a given worker can execute
                the given remote function before it must exit
                (this can be used to address memory leaks in third-party
                libraries or to reclaim resources that cannot easily be
                released, e.g., GPU memory that was acquired by TensorFlow).
                By default this is infinite for CPU tasks and 1 for GPU tasks
                (to force GPU tasks to release resources after finishing).
```

Actor workers are permanently dedicated. Once bound, an actor worker is excluded from the reusable pool via `IsAvailableForScheduling()` ([worker.h#L151-L158](https://github.com/ray-project/ray/tree/master/src/ray/raylet/worker.h))

```cpp
  bool IsAvailableForScheduling() const override {
    return !IsDead()
           && !GetGrantedLeaseId().IsNil()
           && !IsBlocked()
           && GetActorId().IsNil();  // No assigned actor
  }
```

The `actor_id_` field locks worker identity in C++: if nil, the worker executes only stateless tasks ([core_worker.h#L1034-L1037](https://github.com/ray-project/ray/tree/master/src/ray/core_worker/core_worker.h))

```cpp
  ActorID GetActorId() const {
    absl::MutexLock lock(&mutex_);
    return actor_id_;
  }
```

A worker's type is fixed at lease-grant time and cannot switch ([worker.h#L229-L232](https://github.com/ray-project/ray/tree/master/src/ray/raylet/worker.h))

```cpp
  std::optional<bool> is_gpu_ = std::nullopt;
  /// Whether this worker can hold an actor. Once it holds an actor or a normal lease, it
  /// can't switch to the other type.
  std::optional<bool> is_actor_worker_ = std::nullopt;
```

## 2. Resource Reservation: Per-Call vs. Lifetime

Tasks hold resources only while executing. Actors reserve them for their entire lifecycle.

This is implemented via two fields on the worker: `allocated_instances` (flushed per task) and `lifetime_allocated_instances` (held for the actor's lifetime) ([worker.h#L104-L131](https://github.com/ray-project/ray/tree/master/src/ray/raylet/worker.h))

```cpp
  void SetAllocatedInstances(...) override { ... };
  std::shared_ptr<TaskResourceInstances> GetAllocatedInstances() override { ... };

  void SetLifetimeAllocatedInstances(...) override { ... };
  std::shared_ptr<TaskResourceInstances> GetLifetimeAllocatedInstances() override { ... };
```

The lease-grant path routes resources into the correct field by checking whether the spec is an actor creation task ([worker.h#L135-L145](https://github.com/ray-project/ray/tree/master/src/ray/raylet/worker.h))

```cpp
  void GrantLease(const RayLease &granted_lease) override {
    // ...
    SetIsActorWorker(lease_spec.IsActorCreationTask());
    // ...
  }
```

`TaskSpecification` distinguishes `GetRequiredResources()` (execution) from `GetRequiredPlacementResources()` (actor lifetime scheduling) ([task_spec.h#L247-L256](https://github.com/ray-project/ray/tree/master/src/ray/common/task/task_spec.h))

```cpp
  /// Return the resources that are required for a task to be placed on a node.
  /// They may differ from execution resources, e.g., actor creation tasks may
  /// require more resources for scheduling than their methods do at runtime.
  const ResourceSet &GetRequiredPlacementResources() const;
```

Collapsing these two models would make the scheduler blind to allocation lifetime.

## 3. Fault Tolerance: Lineage Reconstruction vs. Process Checkpointing

Because tasks are stateless, a lost output triggers lineage reconstruction. `TaskManager` re-executes the pinned task spec ([task_manager.h#L175](https://github.com/ray-project/ray/tree/master/src/ray/core_worker/task_manager.h)) — object recovery is handled separately by `ObjectRecoveryManager` in `object_recovery_manager.h`

```cpp
class TaskManager : public TaskManagerInterface {
 public:
  std::vector<rpc::ObjectReference> AddPendingTask(...) override;
  std::optional<rpc::ErrorType> ResubmitTask(...) override;
```

`max_retries` is a pure attempt budget ([remote_function.py#L228-L232](https://github.com/ray-project/ray/tree/master/python/ray/remote_function.py))

```python
                crashes unexpectedly. The minimum valid value is 0,
                the default is 3 (default), and a value of -1 indicates
                infinite retries.
```

Actors cannot use lineage reconstruction because their state is the product of sequentially mutating method calls. When an actor crashes, Ray restarts it by re-running `__init__`, erasing all intermediate state. `max_restarts` controls how many cold restarts are permitted ([actor.py#L1638-L1643](https://github.com/ray-project/ray/tree/master/python/ray/actor.py))

```python
            max_restarts: This specifies the maximum
                number of times that the actor should be restarted when it dies
                unexpectedly. The minimum valid value is 0 (default),
                which indicates that the actor doesn't need to be restarted.
                A value of -1 indicates that an actor should be restarted
                indefinitely.
```

`max_restarts` provides process-level continuity, not durable state fault tolerance.

## 4. Free Node Scheduling vs. Strict Dispatch Binding

Tasks are routed to any available node satisfying their resource requirements. Actors must route all method calls to the same dedicated worker. `SubmitTask()` and `SubmitActorTask()` are separate operations in `CoreWorker` ([core_worker.h#L857-L867](https://github.com/ray-project/ray/tree/master/src/ray/core_worker/core_worker.h))

```cpp
  std::vector<rpc::ObjectReference> SubmitTask(
      const RayFunction &function,
      const std::vector<std::unique_ptr<TaskArg>> &args,
      const TaskOptions &task_options,
      int max_retries, ...);
```

([core_worker.h#L932-L961](https://github.com/ray-project/ray/tree/master/src/ray/core_worker/core_worker.h))

```cpp
  Status SubmitActorTask(const ActorID &actor_id,
                         const RayFunction &function,
                         const std::vector<std::unique_ptr<TaskArg>> &args,
                         ...
                         std::vector<rpc::ObjectReference> &task_returns, ...);
```

`SubmitTask()` selects any backend. `SubmitActorTask()` requires an `ActorID` to bind the payload to a specific worker.

## 5. Execution Ordering: Unordered vs. Sequenced

Tasks execute out-of-order to maximize throughput. Actor state integrity requires serialized execution.

Ray enforces FIFO ordering for actor methods via `OrderedActorTaskExecutionQueue`, keyed on a per-concurrency-group sequence counter exposed by `ConcurrencyGroupSequenceNumber()` at line 324 ([task_spec.h#L324](https://github.com/ray-project/ray/tree/master/src/ray/common/task/task_spec.h))

```cpp
  // Methods specific to actor tasks.
  ActorID ActorId() const;
  TaskID CallerId() const;
  // ...
  uint64_t ConcurrencyGroupSequenceNumber() const;
```

([ordered_actor_task_execution_queue.h#L1-L40](https://github.com/ray-project/ray/tree/master/src/ray/core_worker/task_execution/ordered_actor_task_execution_queue.h))


Applying this ordering infrastructure to stateless tasks would destroy parallelism.

## Summary

`TaskSpecification` uses three boolean discriminators — `IsNormalTask()`, `IsActorCreationTask()`, `IsActorTask()` — to fork the execution pipeline throughout the runtime ([task_spec.h#L284-L291](https://github.com/ray-project/ray/tree/master/src/ray/common/task/task_spec.h))

```cpp
  bool IsNormalTask() const;
  bool IsActorCreationTask() const;
  bool IsActorTask() const;
```

Allowing task state destroys the safety and scalability of the `IsNormalTask()` branch.
